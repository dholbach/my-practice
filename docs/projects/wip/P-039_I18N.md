## Goal

Make all user-facing UI text translatable via Django's i18n framework so the app
can run in German or English (or any future language) without code changes.

## Contributor workflow (scavenger hunt)

Once the scaffold is merged, each template becomes a self-contained issue:

1. Add `{% load i18n %}` to the template (alongside `{% load static %}`)
2. Wrap UI strings: `"L√∂schen"` ‚Üí `{% trans "L√∂schen" %}`
3. For strings with variables: use `{% blocktrans %}‚Ä¶{% endblocktrans %}`
4. Run `./dev.py i18n` ‚Äî this extracts strings into `locale/de/django.po`
   and `locale/en/django.po`, then compiles them
5. Add the English `msgstr` values in `locale/en/LC_MESSAGES/django.po`
6. Run `./dev.py i18n compilemessages` again, test in browser

See `inquiry_confirm_delete.html` as the reference implementation.

## Notes

- **New files are never exempt.** The "remaining scope" table above describes the backlog of *pre-existing* files still to migrate ‚Äî it is not a license to write a new template/view/form with unwrapped strings and add it to the backlog. Any file created from this point on must ship fully wrapped (English msgids, `{% trans %}`/`gettext_lazy`) in its first commit. See the "i18n Conventions (P-039)" section in the root `CLAUDE.md` for the exact rules.
- German is the primary language ‚Äî DE strings stay as-is in templates; EN needs translation in `.po` files
- `.po` source files are committed; `.mo` compiled files are generated locally (gitignored)
- `--no-wrap` flag on `makemessages` keeps long strings on one line (easier diffs)
- Language switching uses `Accept-Language` header via `LocaleMiddleware` ‚Äî no URL prefix needed
- `set_language` view at `/i18n/set_language/` available for an explicit language switcher if wanted later
