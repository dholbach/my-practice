# P-039: Django i18n — Bilingual App UI

**Status**: TODO
**Priority**: Low (post-OSS, or driven by user demand)
**Depends on**: P-038 (Language EN cleanup), P-024 (OSS Release)

## Goal

Make all user-facing UI text translatable via Django's i18n framework so the app
can run in German or English (or any future language) without code changes.

## Scope

| Area | Work |
|------|------|
| Django settings | `USE_I18N = True`, `LANGUAGE_CODE`, `LocaleMiddleware` |
| Templates (~50) | `{% load i18n %}` + `{% trans %}` / `{% blocktrans %}` on all UI strings |
| Python — model fields | `verbose_name`, `verbose_name_plural`, `help_text` → `_()` wrappers |
| Python — views | `messages.success()`, `messages.error()` → `_()` wrappers |
| Python — forms | field labels, help_text → `_()` wrappers |
| Python — admin | `list_display` labels, fieldset names → `_()` wrappers |
| Translation files | `locale/de/LC_MESSAGES/django.po` (DE, primary) |
|  | `locale/en/LC_MESSAGES/django.po` (EN translation) |
| UI | Language switcher (per-user or per-session preference) |

## Not in scope

- Email body templates (these are copy-paste text, already have DE/EN variants)
- PDF templates (language is per-invoice, already handled by `invoice_pdf_de/en.html`)
- URL slugs (already English after P-038)

## Rough effort

~2 weeks for a thorough pass. Template extraction alone is ~1 day; translation file
maintenance is ongoing.

## Notes

- German is the primary language; DE strings stay as-is, EN needs translation.
- `makemessages` / `compilemessages` workflow — add to `./dev.py` as `./dev.py i18n`.
- Consider `django-rosetta` for in-browser translation editing.
- `verbose_name` values do NOT need to change before this project — leave them as
  German strings so they are easy to find and wrap in one pass.
